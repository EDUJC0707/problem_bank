# PRD: 교과서·성취기준 통합 RAG 검색 서비스

> **한 줄 요약:** 교과서 5종 PDF, 교육부 성취기준 등 여러 내부 문서를 하나의 RAG 시스템에 넣고, LLM에게 자연어로 질문하면 관련 정보를 근거(출처)와 함께 답하고, 클릭 한 번으로 **해당 PDF의 정확한 페이지를 열어 하이라이트**해 주는 서비스.

- 문서 버전: v0.2 (초안)
- 작성일: 2026-07-07
- 작성자: (담당자)
- 상태: Draft — 검토 대기
- 확정된 전제: **사내 소수 사용자용**, **외부 LLM API(클라우드) 호출 허용**

---

## 1. 배경 & 문제 정의

### 1.1 배경
교사·연구원·콘텐츠 제작자는 수업 준비, 문항 제작, 교육과정 정합성 확인을 위해 여러 출판사 교과서와 교육부 성취기준 문서를 **동시에** 참고해야 한다. 현재는:

- 문서가 PDF 파일로 흩어져 있어 원하는 개념이 **어느 교과서 몇 페이지**에 있는지 일일이 찾아야 한다.
- "이 성취기준을 다루는 단원이 5종 교과서에서 각각 어디냐" 같은 **교차 비교 질의**를 사람이 수작업으로 처리한다.
- 검색 결과를 신뢰하려면 **원문 페이지를 직접 눈으로 확인**해야 하는데, 그 과정이 느리다.

### 1.2 문제
> 여러 내부 교육 문서에 흩어진 정보를 **자연어 질문 한 번**으로 모아 보고, 각 정보가 **어느 문서 몇 페이지에서 나왔는지 즉시 원문으로 확인**할 수단이 없다.

### 1.3 이 제품이 해결하는 것
1. 여러 문서를 하나의 지식베이스로 통합 검색.
2. LLM이 관련된 **모든** 관련 출처를 취합해 답변.
3. 각 근거마다 **출처(교과서명·출판사·단원·페이지) 표기 + 원문 페이지 바로 열기 + 하이라이트**.

## 2. 목표 & 비목표

### 2.1 목표 (Goals)
- **G1.** 자연어 질문에 대해 5종 교과서 + 성취기준 문서에서 관련 근거를 검색·요약해 답변한다.
- **G2.** 모든 답변에 **검증 가능한 출처**를 붙인다(문서명·페이지·인용문). 출처 없는 주장은 하지 않는다.
- **G3.** 출처 클릭 시 해당 PDF의 정확한 페이지로 이동하고 관련 영역을 하이라이트한다.
- **G4.** 교차 문서 질의("5종 교과서에서 광합성을 다루는 단원 비교")를 지원한다.

### 2.2 비목표 (Non-Goals, v1 범위 밖)
- 문서 자동 편집/생성(문항 자동 출제 등) — 별도 제품으로 분리.
- 다중 사용자 권한/조직 관리(RBAC) 고도화 — v1은 단일 팀/소수 사용자 가정.
- 모바일 전용 앱 — v1은 웹(데스크톱 브라우저) 우선.
- 실시간 협업/코멘트 기능.
- 저작권 라이선스 관리 워크플로우(문서 반입은 내부 승인 완료 전제).

## 3. 대상 사용자 & 핵심 시나리오

### 3.1 페르소나
- **P1. 교사/교육 콘텐츠 제작자** — 수업·문항 준비 중 개념의 교과서별 서술과 근거 페이지를 빠르게 찾고 싶다.
- **P2. 교육과정 연구원** — 성취기준과 교과서 단원의 정합성을 교차 확인하고 싶다.

### 3.2 대표 사용자 시나리오
1. **개념 근거 찾기**: "‘일과 에너지’ 단원에서 운동 에너지 공식을 설명한 부분 찾아줘" → 답변 + 3개 교과서의 해당 페이지 링크.
2. **성취기준 매핑**: "성취기준 [9과07-03]을 다루는 교과서 단원들 알려줘" → 성취기준 원문 + 각 교과서 매핑 페이지.
3. **교차 비교**: "광합성의 명반응을 5종 교과서는 각각 어떻게 설명해?" → 출판사별 서술 요약 + 페이지.
4. **원문 확인**: 답변의 각 근거 카드 클릭 → 우측(또는 새 탭) PDF 뷰어에서 해당 페이지 열림 + 하이라이트.

## 4. 기능 요구사항

### 4.1 문서 수집 & 인덱싱 (Ingestion)
- **F1.1** PDF 업로드(개별/일괄). 문서별 메타데이터 입력: 문서 유형(교과서/성취기준/기타), 출판사, 과목, 학년, 제목.
- **F1.2** PDF 파싱: 페이지 단위 텍스트 추출 + **페이지 번호 및 텍스트 위치(bounding box) 보존**(하이라이트에 필요).
- **F1.3** 스캔본 대비 OCR 폴백(텍스트 레이어 없는 PDF 감지 시).
- **F1.4** 표·이미지 처리: v1은 표를 텍스트로 변환, 이미지는 캡션/주변 텍스트로 인덱싱(이미지 자체 이해는 v2).
- **F1.5** 청킹: 페이지 경계를 인지하는 의미 단위 청킹. 각 청크는 **출처 메타데이터(문서ID·페이지·위치·단원·성취기준코드)** 를 유지.
- **F1.6** 임베딩 생성 후 벡터 저장소 + 키워드 인덱스에 동시 색인(하이브리드 검색용).
- **F1.7** 재색인/문서 삭제·교체 지원.

### 4.2 검색 & 답변 (Retrieval + Generation)
- **F2.1** 하이브리드 검색: 벡터 유사도 + 키워드(BM25) 결합, 리랭킹으로 상위 근거 선별.
- **F2.2** 메타데이터 필터: 출판사/과목/학년/문서유형으로 범위 제한.
- **F2.3** LLM이 검색된 근거만을 바탕으로 답변 생성(근거 밖 내용 생성 억제, 모르면 "관련 근거 없음"으로 응답).
- **F2.4** 답변 내 각 문장·주장에 **출처 인용 번호** 부착.
- **F2.5** 근거 카드 리스트: 문서명·출판사·단원·페이지·인용 스니펫 표시. 관련도 순 정렬.
- **F2.6** 교차 문서 질의 시 문서별로 그룹화해 제시.

### 4.3 출처 표기 & 페이지 열기 (핵심 차별점)
- **F3.1** 답변의 인용 번호 ↔ 근거 카드 ↔ 원문 위치가 1:1로 연결.
- **F3.2** 근거 카드 클릭 시 내장 PDF 뷰어에서 **해당 페이지로 즉시 이동**.
- **F3.3** 이동 후 인용에 해당하는 텍스트 영역을 **하이라이트**(bounding box 기반).
- **F3.4** 뷰어에서 앞뒤 페이지 탐색, 확대/축소, 원본 다운로드(권한 시).

### 4.4 검색 히스토리 (선택, v1 경량)
- **F4.1** 최근 질문/답변 목록 저장 및 재열람.

```
│  필터: [출판사▾][과목▾]     │   │  A출판사 과학 3       │ │

│  ── 답변 ─────────────     │   │  ┌─────────────────┐  │ │

│                            │   │                       │ │
│  ── 근거(출처) ──────       │   │  [◀ p.127] [p.129 ▶]  │ │

```

- **UI-1** 단일 질문 입력창 + 전송, 상단에 메타데이터 필터.
- **UI-2** 답변 영역: 인용 번호 클릭 시 해당 근거 카드로 스크롤/하이라이트.
- **UI-3** 근거 카드 리스트: 클릭 시 우측 PDF 뷰어에 원문 페이지 로드 + 하이라이트.

- **UI-5** 문서 관리 화면(별도): 업로드·메타데이터 입력·재색인·삭제.

## 6. 시스템 아키텍처 (개략)

```
[문서 업로드] → [PDF 파서 + OCR] → [청킹 + 메타데이터] → [임베딩]
                                                          │
                          ┌───────────────────────────────┤
                          ▼                               ▼
                   [벡터 저장소]                    [키워드 인덱스]
                          │                               │
[사용자 질문] → [하이브리드 검색 + 리랭킹] ◀──────────────┘
                          │
                          ▼
        [LLM 답변 생성 (근거 기반, 출처 인용)]
                          │
                          ▼
        [답변 + 근거카드 + 페이지/좌표 → PDF 뷰어]
```

### 6.1 기술 스택 (사내 소수 · 외부 API 허용 기준 확정안)
- **생성 LLM:** **Claude(Anthropic API)**.
  - 기본: **Claude Opus 4.8**(`claude-opus-4-8`) — 최고 품질, 1M 토큰 컨텍스트, 한국어·문서이해 강점. 사용자 수가 적어 비용 부담이 크지 않음.
  - 비용/속도 최적화 대안: **Claude Sonnet 5**(`claude-sonnet-5`) — 품질 대비 저비용(도입가 프로모션 적용 중). 라우팅으로 혼용 가능.
  - 스캔본 대비: Claude는 **PDF 내 이미지·스캔 페이지를 네이티브 비전으로 판독** → 별도 OCR 의존도를 낮출 수 있음(품질 검증 후 결정).
- **임베딩(검색용 벡터):** **별도 제공자 필요** — Anthropic은 임베딩 모델을 제공하지 않음.
  - 후보: Voyage AI(멀티링구얼), OpenAI 임베딩, 또는 사내 온프레미스 오픈소스(BGE-M3 등). **한국어 성능**을 1순위 기준으로 선정.
- **벡터 저장소:** 관리형/오픈소스 벡터 DB 중 택1(문서 수십~수백 개 · 소수 사용자 규모라 운영 부담 낮은 것 우선).
- **문서 파싱:** PDF 텍스트 추출 라이브러리(+ 필요 시 좌표/스캔 OCR). 아래 6.2의 인용 방식 채택 시 좌표 저장은 선택 사항.
- **프런트엔드:** 웹(React 등) + 브라우저 내 PDF 렌더러(PDF.js 등)로 페이지 이동·하이라이트.
- **백엔드:** 검색(하이브리드)·리랭킹·LLM 호출 오케스트레이션 API 서버.

> **규모 가정(확정): 문서 수십~수백 개, 동시 사용자 소수(<50).** 이 규모에서는 관리형 서비스 위주로 구성해 운영 부담을 최소화.

### 6.2 출처 표기 — Claude 네이티브 Citations로 환각 방지 (핵심 설계)

이 제품의 최대 리스크는 "LLM이 그럴듯하지만 틀린 출처/페이지를 지어내는 것"이다. 이를 구조적으로 차단하기 위해 **Claude API의 내장 Citations 기능**을 사용한다.

- **동작:** 검색으로 뽑은 근거 청크를 문서 블록으로 전달하고 `citations` 를 활성화하면, Claude는 답변 문장마다 **실제로 인용한 원문 텍스트(cited_text)와 위치(페이지 번호 등)를 API 응답으로 반환**한다. 출처를 LLM이 "서술"하는 게 아니라 **API가 근거와 매핑해 돌려주므로 환각이 원천 차단**된다.
- **페이지 매핑:** 각 근거 청크에 원본 **문서ID·페이지 번호**를 메타데이터로 유지 → 반환된 인용을 그 페이지로 즉시 연결.
- **하이라이트 단순화:** 반환된 **cited_text 문자열**을 PDF 뷰어(PDF.js)에서 텍스트 검색·강조하는 방식이면, **픽셀 좌표(bbox) 저장 없이도** 하이라이트 구현 가능 → M0 스파이크 난이도와 파싱 파이프라인이 크게 단순해짐. (스캔·이미지 페이지 등 텍스트 레이어가 없을 때만 좌표 기반 폴백.)

> 결론: **"어느 페이지·어떤 문장"의 근거 확정은 Citations API가, 화면상의 시각적 강조는 PDF 뷰어의 텍스트 검색이 담당**. 이 조합이 F2.4/F3.1~F3.3(출처 인용·페이지 열기·하이라이트)을 가장 안전하게 충족한다.

## 7. 데이터 모델 (핵심 엔티티)

- **Document**: id, 유형(textbook/standard/other), 출판사, 과목, 학년, 제목, 파일경로, 페이지수, 색인상태.
- **Chunk**: id, document_id, 페이지번호, 텍스트, bounding_box[], 단원, 성취기준코드[], 임베딩벡터.
- **Query/Answer(선택)**: id, 질문, 답변, 인용된 chunk_id[], 생성시각.

## 8. 비기능 요구사항

- **정확성:** 출처 없는 주장 금지. 근거가 부족하면 "관련 근거 없음" 명시. (RAG 환각 억제)
- **응답 속도:** 일반 질의 체감 응답 목표 ~5초 내(스트리밍 시작 기준).
- **보안/저작권:** 교과서는 저작물. **사내 인증 사용자만 접근**(소수 규모라 단순 인증 + 접근 로그로 충분), 문서 반출·다운로드 통제, 외부 유출 방지.
- **외부 LLM API 데이터 처리(확정 사항 기반):** 클라우드 LLM 호출을 허용하므로, 질문·검색된 근거 청크가 API로 전송됨. 다음을 준수:
  - 제공자와 **데이터 미학습(Zero/No-training) 조건** 및 데이터 보존 정책 확인·계약.
  - 전송 대상은 **검색으로 선별된 근거 청크**로 최소화(문서 전체 상시 업로드 지양).
  - API 키는 서버 측 시크릿으로 관리(클라이언트 노출 금지).
- **개인정보:** 질문 로그에 개인정보 최소 수집.
- **확장성:** 문서 추가/교체가 운영 중 가능(무중단 재색인 목표는 v2).

## 9. 성공 지표 (KPI)

- **채택:** 주간 활성 사용자, 사용자당 질문 수.
- **품질:** 답변에 유효 출처가 붙은 비율(≥95% 목표), 사용자 "도움됨" 평가율.
- **효율:** 사용자가 원하는 근거 페이지에 도달하기까지 시간(기존 수작업 대비 단축률).
- **신뢰:** 잘못된/무근거 답변 신고율(낮을수록 좋음).

1. **배포 대상 = 사내 소수 사용자.** → 인증·인프라를 경량으로, 관리형 서비스 중심 구성.

### 남은 오픈 질문 (확정 필요)
